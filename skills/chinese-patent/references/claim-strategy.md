# 权利要求布局策略详解

> 专利的价值不在说明书，在权利要求。权利要求写得好，竞争对手绕不过去；写得差，形同废纸。

---

## 一、洋葱式防线：核心策略

### 1.1 为什么重要

专利被无效时，通常从独立权利要求开始攻击。如果权1被无效，权2-8如果只是权1的简单细化，那么整条防线一起崩。

**好的布局**：即使权1被无效，从属权利要求中还有独立的技术方案可以"上位"替代。

### 1.2 布局模板

```
权1（独立）── 最少必要特征，最宽保护范围
├─ 权2 ── 优选结构特征A（常见实施方式）
│   └─ 权3 ── 参数范围/具体配置
├─ 权4 ── 优选结构特征B（替代实施方式）← 关键防线
│   └─ 权5 ── B的细化
├─ 权6 ── 方法步骤的具体限定
└─ 权7 ── 存储介质

权8（独立）── 系统/装置权利要求
├─ 权9 ── 模块细化
└─ 权10 ── 存储介质
```

### 1.3 关键原则

**原则1：权1能少则少**
- 只放解决技术问题的**最少必要特征**
- 多放一个特征 = 保护范围缩小一圈
- 被砍掉的特征 → 全部变成从属权利要求

**原则2：替代方案写入从属权利要求**
- 如果核心创新可以用方式A或方式B实现，权1写两者共有的上位概念
- 方式A → 权2，方式B → 权4
- 这样即使方式A被现有技术否定，方式B还在

**原则3：参数范围分层**
- 权1：不限定具体参数（或用"至少一个""多个"等模糊但合规的表述）
- 权2：限定一个宽范围（如"3-10层"）
- 权3：限定最优范围（如"3-5层"）

### 1.4 示例

**差的权利要求布局：**
```
权1: 一种图像分类方法，包括：提取RGB和红外图像的ResNet-50特征；
     使用3层交叉注意力进行融合；使用softmax分类。
```
→ 问题：限定死了 ResNet-50、3层、RGB+红外，保护范围极窄。

**好的权利要求布局：**
```
权1: 一种多模态图像分类方法，包括：提取至少两种模态图像的特征向量；
     以第一模态特征为查询向量、第二模态特征为键值向量进行交叉注意力融合；输出分类结果。
→ 没有限定具体网络、层数、模态类型，保护范围最宽。

权2: 根据权1，特征提取采用深度神经网络。
权3: 根据权2，深度神经网络为卷积神经网络。
权4: 根据权3，卷积神经网络为ResNet-50。
权5: 根据权1，交叉注意力融合包含至少两层级联。
权6: 根据权5，层数为3层。
→ 即使权4被无效（ResNet-50是现有技术），权3还能保护用其他CNN的方案。
→ 即使权5-6被无效，权1还能保护用1层交叉注意力的方案。
```

## 二、侵权规避视角

### 2.1 核心问题

写权利要求时，不要只从"我怎么实现"的角度写。要从"别人怎么绕过去"的角度写。

### 2.2 具体技巧

**技巧1：用开放式表述**
- ✅ "包括/包含/含有" → 意味着还可以有其他组件
- ❌ "由……组成" → 只能是这些，加一个就不侵权
- 默认使用"包括"，除非有充分理由

**技巧2：功能性描述 > 实现性描述（但注意审查风险）**
- ✅ "配置为对数据进行加密的模块"
- ❌ "使用AES-256算法对数据进行加密的模块"
- CNIPA 注意：纯功能性限定可能被审查员驳回（"得不到说明书支持"），所以要平衡。最好写"配置为[功能]的[模块类型]"，既体现功能又有结构。

**技巧3：步骤顺序不限定（如果不必须）**
- 如果方法步骤A和B的顺序可以互换，不要写"先A后B"
- 写"A和B"而不是"先执行A，再执行B"

**技巧4：数量不限定（如果不必须）**
- ✅ "多个"而不是"3个"
- ✅ "至少一个"而不是"一个"
- 在从属权利要求中再限定具体数量

**技巧5：预留替代方案**
- 如果区别特征是"交叉注意力"，在从属权利要求中加"自注意力""混合注意力"等替代
- 这样如果交叉注意力被判定为现有技术，替代方案可能还有价值

### 2.3 侵权难度自测

每写完一条权利要求，做以下测试：

> **测试1：替换测试** — 如果竞争对手把[特征X]换成[相似特征Y]，是否还能解决同样的技术问题？
> 如果能 → 特征X写得太窄，考虑抽象一层。

> **测试2：删减测试** — 如果竞争对手直接省略[特征Z]，是否还能达到[部分]技术效果？
> 如果能 → 特征Z可能不应该放在独立权利要求中。

> **测试3：组合测试** — 如果竞争对手把两个特征合为一个来实现，是否落入保护范围？
> 如果否 → 考虑增加"或者其等效方式"类表述（但注意 CNIPA 对此的审查态度）。

## 三、常见致命错误

### 3.1 前序部分写太窄
```
❌ 一种基于ResNet的多模态图像分类方法...
✅ 一种多模态数据处理方法...
```
前序部分的主题名称会限定整条权利要求的范围。

### 3.2 必要特征缺失
如果独立权利要求缺少一个解决技术问题必须的特征，审查员会以"缺少必要技术特征"驳回。
→ 解决：确保权1包含了所有"没有它就解决不了技术问题"的特征。

### 3.3 从属权利要求没有退守价值
如果所有从属权利要求都只是在权1上加细节，权1被无效后全部跟着死。
→ 解决：确保有至少一条从属权利要求包含替代实施方案。

### 3.4 说明书与权利要求脱节
权利要求中出现的术语在说明书中没有解释或用法不同。
→ 解决：使用特征-支持交叉引用表强制检查。

### 3.5 软件/AI 专利缺少技术效果
纯算法描述没有关联到技术效果，被审查员以"不是技术方案"驳回。
→ 解决：确保技术问题和技术效果都是"技术的"（效率、速度、资源、精度），不是"商业的"。
